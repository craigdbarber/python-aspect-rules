How to Finally Fix "Works on My Machine" with a Hermetic Python Blueprint

By Craig Barber • May 6, 2026

Let's be honest: Python environment management can be a bit of a nightmare. We've all had that moment where a script runs perfectly on our laptop, only to fall apart the second it hits a CI runner or a teammate's machine. Whether it's a missing system library, a slightly different Python minor version, or a transitive dependency that decided to change under our feet, "environment drift" is the silent killer of productivity.

That's exactly why I built the Bazel Python Hermetic Blueprint. It’s not just a demo project; it’s a production-ready architectural template designed to bring real industrial rigor to Python development. We’re talking about builds that are truly isolated, reproducible, and fast.

The Big Idea: Why True Hermeticity?

In the world of build systems, "hermetic" is a fancy way of saying "completely isolated." A hermetic build doesn't care if you have Python 3.10 or 3.12 installed on your system. It doesn't care what libraries are in your /usr/local/lib. It brings its own tools to the party.

By using Bazel as our core, we've pinned every single layer of the stack:

  • The Interpreter: Bazel downloads the exact Python 3.14 binary for you. Everyone on the team uses the same one.
  • The Toolchains: Your linters, compilers, and formatters are all version-locked and isolated.
  • The Dependencies: We use uv to resolve packages and verify them with SHA-256 hashes. If the bytes don't match the lockfile, the build fails.
The Bottom Line: If it builds on my Mac, it builds on your Linux server. No exceptions.

Why Bazel? (The Superpowers)

You might ask: "Why use Bazel when I have pip and virtual environments?" The answer lies in how Bazel handles the build graph. Unlike traditional Python tools, Bazel treats every file and dependency as a node in a massive, verifiable mathematical equation. This gives us several "superpowers":

  • Correctness: Bazel tracks every single input to a command. If a file hasn't changed, it won't rebuild it. If it has, it guarantees that everything depending on it is updated.
  • Massive Parallelism: Because Bazel understands exactly which tasks are independent, it can run your tests, audits, and builds in parallel across all your CPU cores without any race conditions.
  • Remote Caching: In a team setting, if one person builds a target, everyone else can download the pre-built result instead of compiling it themselves.
  • Language Agnostic: While this is a Python blueprint, Bazel can manage C++, Rust, or Go in the same graph, making it the perfect choice for polyglot projects.

A Stack Built for Zero-Debt Quality

Shipping features is great, but shipping clean features is better. The blueprint includes a multi-layered quality control system that runs every time you build.

1. Dependency Mastery with uv

We're using uv, which is basically the fastest Python resolver ever made. By plugging it into Bazel, we get lightning-fast package resolution combined with Bazel's rock-solid reproducibility. It's the best of both worlds.

2. "Set It and Forget It" Automation

I hate "formatting debt"—those annoying PR comments about missing spaces or inconsistent quotes. To fix this, we use a tools/bazel wrapper. It automatically runs the Ruff formatter whenever you build or test. The cool part? It's smart enough to stay read-only in CI, so your automated checks remain pure.

3. Full-Spectrum Auditing

  • Logic: Pylint does the heavy lifting for deep code analysis.
  • Security: Bandit is baked in to catch unsafe patterns (like hardcoded keys or eval calls) before they ever hit production.
  • Documentation: We even lint our Markdown! Vale ensures your READMEs are as professional as your code.

Putting It to Work (Examples)

One of the best things about Bazel is that the interface is remarkably consistent. Here’s how you actually interact with the blueprint day-to-day:

Running the App

bazel run //:main

Running the Test Suite

bazel test //:test_lib

The "Deep Scan" Security Audit

bazel test //tools/lint:bandit_test

Checking Your Docs

bazel test //tools/lint:markdown_lint_test

The "Everything" Audit

Run style, logic, and security checks across the whole project at once:

bazel build --config=lint //...

Adopting the Blueprint Step-by-Step

Ready to try it out? Here’s the quickest way to get this architecture running for your own project:

  1. Grab the foundation: Clone the repo and strip the history to make it your own. 
    git clone https://github.com/craigdbarber/bazel-python-hermetic-blueprint.git my-project
cd my-project
rm -rf .git && git init
  2. Rename your project: Just pop into MODULE.bazel and pyproject.toml and update the names.
  3. Drop in your code: Move your logic into src/ and your tests into tests/. Link them up in BUILD.bazel.
  4. Add your packages: Use uv to get your dependencies in order. 
    uv add pandas fastapi
uv lock
  5. Verify everything: Run the audit command. If it passes, you know your environment is solid. 
    bazel build --config=lint //...

When Should You Use This?

You might think this is "overkill" for a small project, but the benefits scale down just as well as they scale up:

  • Small Projects & Side Hustles: Even if it's just you, this blueprint ensures that your project stays healthy over months of inactivity. You won't come back to a project and find that your local pip install no longer works because of a new OS update. It's "future-proofing" your code.
  • Scaling Monorepos: Keep dozens of teams consistent without the headache.
  • Regulated Industries: When you need to prove exactly what's in your binaries and audited by hash.
  • CI-Heavy Workflows: Leverage Bazel's caching to turn 20-minute CI runs into 20-second ones.

Wrapping Up

The Bazel Python Hermetic Blueprint is about more than just build rules. It's about shifting your focus from "how do I get this environment to work?" to "how do I ship this feature?" By embracing hermeticity and automated audits, you're building a foundation that can scale as far as your code needs to go.
